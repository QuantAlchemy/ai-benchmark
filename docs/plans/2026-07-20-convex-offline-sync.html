AI Benchmark Convex Offline Sync Implementation Plan

Goal: Make benchmark runs and complete generated solutions available across local clones while preserving durable offline operation and automatic synchronization through Convex.

Architecture: Each OS user keeps a private SQLite replica and durable outbox. A dedicated Convex production deployment stores authenticated, append-only run snapshots/events and chunked solution artifacts in Convex File Storage. Launch and Verify automatically hydrate missing solutions from the local artifact cache or Convex.

Tech stack: Node.js 24+, TypeScript, node:sqlite, Convex 1.42.3, Convex File Storage, Vitest, tar archives, SHA-256, React Router/TanStack Start.

Execution: Use strict RED–GREEN–REFACTOR for production behavior. Run a spec-compliance review and code-quality/security review before the final Quant Alchemy quality gate and PR.

Acceptance criteria

  1. A run created without network access survives app restart in local SQLite.
  2. Reconnect pushes queued operations idempotently to Convex.
  3. A second independent replica pulls the run and complete solution artifact.
  4. After disconnecting again, the second replica can materialize and launch the synchronized solution.
  5. Launch and Verify hydrate missing paths automatically rather than requiring rsync.
  6. Multiple users on one host are distinguished by stable client UUIDs and private aliases.
  7. Convex endpoints reject unauthenticated clients; secrets and identities are absent from Git.
  8. Archive creation excludes credentials/caches and extraction rejects traversal, unsafe links, corruption, and size violations.
  9. Existing SQLite rows migrate idempotently and remain usable.

Data and conflict model

  • Identity: opaque client UUID plus private alias; no automatic raw username/hostname upload.
  • Runs: stable run_uid in addition to the local numeric primary key.
  • Operations: application-generated UUIDs and idempotent server insertion.
  • Artifacts: immutable content digest, manifest, fixed-size chunks, and per-chunk digest.
  • Changes: append-only server sequence for reliable cursored pull.
  • Deletes: tombstone events rather than immediate remote destruction.
  • Mutable fields: deterministic event order with all source events retained for audit.

Security gates

  • All sync HTTP actions require a per-client bearer token stored only as a hash in Convex.
  • Administrative provisioning requires a separate secret stored in Convex environment variables and 1Password.
  • Browser code never receives deployment keys, admin secrets, or client credentials.
  • Direct Convex storage URLs are not exposed; authenticated routes serve bounded chunks.
  • Bundle paths are relative, normalized, and allowlisted; .env*, .git, dependency caches, auth files, sockets, and device files are excluded.
  • Extraction uses a temporary directory, validates every member, verifies hashes and limits, then atomically renames.

Deployment and validation

  1. Deploy to the dedicated private synchronization production deployment.
  2. Provision approved installation identities and store their client credentials plus the admin secret in a private credential manager.
  3. Run integration validation with two isolated data roots and client credentials.
  4. Disable network access to the sync endpoint and prove local reads, writes, restart survival, materialization, and launch.
  5. Restore access and prove automatic convergence and zero pending outbox records.
  6. Export/backup the Convex deployment after successful validation.
